§ 01 / The Premise

A shared archive for course knowledge

Browse and search documents, comment in nested threads (with PDF page-pinned notes, @mentions and reactions), vote on a request board, and manage per-user storage quotas — all gated by a role & review model so only vetted material surfaces publicly.

S

Student

Contributor · enrolled
  • Uploads only to enrolled courses
  • Submissions forced to draft, routed for review
  • Favorites, requests, study collections
  • Sees only own hidden drafts
L

Lecturer

Reviewer · per-course
  • Uploads to courses they teach
  • Approve / reject from the review queue
  • Per-course analytics dashboard
  • Has a pending-approval state
A

Admin

Moderator · global
  • Combined Admin Approvals queue
  • Full-corpus visibility & insights
  • User management & orphaned files
  • Moderation-only: no Upload surface
§ 02 / The Engine Room

Nine systems doing the heavy lifting

01
Search v2 · single source of truth

Prefix, fuzzy & faceted full-text search

Partial words match (lect → lecture) via to_tsquery prefixing, with a trigram word_similarity fallback so typos still land (plankron → plankton). The haystack spans titles, descriptions, course meta, tags, filenames, uploader names and extracted file text.

Discovery

Engagement-aware ranking

A deterministic weighted score: ln-dampened engagement + recency half-life + metadata-completeness, read from denormalised counters — no per-request GROUP BY.

Diagram E — Upload & signed-URL flow through GCS
Client POST /api/documents/upload (multipart). Server magic-byte sniffs the MIME, enforces MAX_UPLOAD_MB & the user's storage quota.
sha256 dedup — a checksum-equivalent file the user already owns short-circuits to the existing doc instead of re-uploading.
The storage adapter writes bytes — either to a GCS bucket mounted as a volume, or natively via @google-cloud/storage. A new versioned DocumentFile row is recorded.
Extraction chain pulls text + thumbnails (sharp / pdf-parse); the FTS tsvector is refreshed.
On preview/download the API mints a short-lived signed URL (SIGNED_URL_TTL_SECONDS, default 300s) bound to the doc id + requesting user — the browser fetches the bytes directly.
Two ways to put files in GCS
OPTION A1 · no code change

Mount the bucket as a volume

Cloud Run mounts the GCS bucket as a filesystem volume; keep STORAGE_DRIVER=local pointed at the mount path. The existing filesystem driver reads/writes the bucket transparently.

▶ fastest to demo
OPTION A2 · native adapter

Implement GcsStorageAdapter

Add @google-cloud/storage, implement the adapter (ADC auth = the service account), set STORAGE_DRIVER=gcs + GCS_BUCKET. No volume needed; exercised by the repo's regression:gcs script.

▶ proper long-term fix
Two code facts that block a naive deploy
GOTCHA 01 · storage

The gcs & s3 drivers are throwing stubs

Out of the box only local works — the cloud drivers raise "…is not implemented yet". Use A1's volume mount or implement A2 before flipping STORAGE_DRIVER.

GOTCHA 02 · auth

Session cookie is sameSite: "lax"

Two distinct *.run.app origins break login. Fix: sameSite: "none" in production (two-origin), or front both with one HTTPS load balancer so lax stays valid.

§ 06 / Honest Edges

What it deliberately doesn't do — yet

✕Notifications are polling-only — no websocket push or email digest.
✕No OCR for scanned PDFs — digital text extraction only.
✕Full-text search only — no semantic / vector search.
✕No multi-reviewer approval quorum.
✕Analytics CSV export is on-demand, not scheduled.
✕Learning-path UI deferred — schema is in place.
