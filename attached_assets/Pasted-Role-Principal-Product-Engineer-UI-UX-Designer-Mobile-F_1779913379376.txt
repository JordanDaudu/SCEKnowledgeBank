Part A — UI/UX Enhancement

Use the current app style as the base. Do not replace the whole design system. Improve it.

1. Global layout and navigation

* Ensure authenticated users have a clear top navigation.
* Student users should see role-appropriate actions:

  * Home
  * Browse/Search
  * Upload, if enrolled in at least one course
  * Requests
  * Following/Favorites if available
  * Notifications bell
* Lecturer/admin users should see:

  * Home
  * Browse/Search
  * Upload
  * Review Queue
  * Requests
  * Analytics
  * Notifications bell
* Ensure the notification bell is visible for all authenticated users.
* Ensure unread badge works visually and does not break layout.
* Add active route styling consistently.
* Make logout/profile area clean and responsive.

2. Home page polish
   Improve the home page so it feels like a finished dashboard, not only a landing page.

For all roles:

* Reduce excessive hero height.
* Make the main search bar visually stronger and easier to use.
* Add a compact “Quick actions” row with role-based actions.
* Improve spacing between sections.
* Improve section headers and visual hierarchy.

For students:

* Highlight:

  * Continue Reading
  * Latest Additions
  * My Uploads / Pending Review / Rejected uploads if existing endpoints support it
  * Following/Favorites if available
  * Requests
* Make it clear that students can upload materials for review if enrolled.

For lecturers/admins:

* Highlight:

  * Pending Review queue
  * Recent uploads
  * Requests needing attention
  * Analytics shortcut
  * Latest activity if existing endpoints support it

Do not create new backend endpoints unless a very small existing query can be reused safely.

3. Document cards
   Polish document cards used on Home/Browse/Search:

* clearer title hierarchy,
* consistent course badge,
* material type badge or label,
* status badge when relevant,
* updated/uploaded date,
* uploader if available,
* better spacing,

4. Search/Browse page

7. Notifications

  * type/message,

8. Request board
   Polish request board:

* Upvote button should be visually clear.

9. Analytics
   Polish analytics pages:

* Cards/tiles should be consistent.

Part B — Mobile Compatibility

Make the app responsive and usable on mobile widths.

Target breakpoints:

* mobile: 320px–480px
* large phone: 481px–640px
* tablet: 641px–900px
* desktop: 901px+

Required mobile behavior:

1. Navigation

* Top nav should not overflow.
* Use a hamburger/menu drawer or compact mobile nav if needed.
* Brand/logo should remain visible.
* User/profile/logout area should be accessible.
* Notification bell should remain accessible.
* Active route should still be clear.

2. Layout

* Hero section should not consume the entire mobile screen.
* Cards should stack vertically.
* Tables should either:

  * become cards, or
  * scroll horizontally only when unavoidable.
* Forms should use full-width inputs on mobile.
* Modals should fit the viewport and be scrollable if needed.
* Dropdowns should not render off-screen.

3. Touch usability

* Buttons and clickable targets should be at least comfortable touch size.
* Avoid tiny icon-only buttons without labels/tooltips where possible.
* Ensure status chips and action buttons do not overlap.

4. Important mobile screens to test manually
   Check at 375px, 390px, 430px, 768px, and desktop:

* Login
* Home
* Browse/Search
* Document detail
* Upload
* Review Queue
* Notifications dropdown/page
* Requests
* Analytics
* Following/Favorites
* Admin/lecturer-specific pages

5. Accessibility basics

* Keep visible focus states.
* Ensure buttons have accessible labels.
* Ensure icon-only buttons have `aria-label`.
* Ensure color contrast remains readable.
* Ensure forms have labels or accessible names.
* Do not trap keyboard focus incorrectly in modals/dropdowns.
