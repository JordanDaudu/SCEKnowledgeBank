Issue 1 — raw-SQL search visibility still leaks `draft` documents:
The Prisma visibility filter correctly hides `draft`, `pending_review`, and `rejected` from non-uploader/non-owner/non-reviewer users.
But `visibleDocumentFilterSql(user)` in `artifacts/api-server/src/services/permissions.service.ts` still only excludes:

`d.status NOT IN ('pending_review', 'rejected')`

This raw-SQL filter is used by v2 search, facets, and autocomplete. It must also exclude `draft`.

Required fix:

* Update `visibleDocumentFilterSql(user)` so its status predicate matches `visibleDocumentFilter(user)` exactly:

  * public/restricted/private visibility rules remain unchanged,
  * status-hidden set must be `draft`, `pending_review`, `rejected`,
  * uploader/owner may still see their own draft/pending/rejected documents,
  * lecturer reviewers may still see draft/pending/rejected documents for courses they teach,
  * admins still bypass the filter.
* Avoid duplicating hidden statuses in multiple places if possible. Prefer one shared helper/constant or at least make the Prisma and SQL branches visibly consistent.

Issue 2 — notification dedup schema/docs/repo mismatch:
The docs and repository comments still say notifications dedupe on:

`recipientId + type + subjectType + subjectId`

But Prisma currently has:

`@@unique([recipientId, subjectType, subjectId])`

This can suppress valid review notifications in a reject → resubmit → approve cycle, because `document.rejected` and `document.approved` share the same recipient/document subject.

Required fix:
Choose one of these two approaches and make schema, migrations, repository, comments, tests, and `replit.md` consistent:

Preferred approach:

* Restore dedup to `recipientId + type + subjectType + subjectId`.
* Add a migration that drops the current subject-only unique index and recreates it including `type`.
* Update `lib/db/prisma/schema.prisma`.
* Update `notifications.repo.ts` comments.
* Keep producer-level logic responsible for avoiding duplicate mention/reply notifications for the same comment.
* This allows `document.rejected` and `document.approved` to both notify the uploader for the same document across separate review cycles.

Alternative approach:

* Keep subject-only dedup, but change review notification `subjectId` so each review decision event is unique enough to allow both rejection and later approval.
* If using this approach, document exactly why and add tests proving reject → resubmit → approve sends both notifications.

Required tests for Issue 2:

* A notification insert with same recipient/type/subject should dedupe.
* A notification insert with same recipient/subject but different type should be allowed, unless using the alternative event-specific subjectId design.
* Review flow test:

  * reject a pending document,
  * resubmit it,
  * approve it,
  * assert both `document.rejected` and `document.approved` notification writes are attempted and not suppressed by repo semantics.
